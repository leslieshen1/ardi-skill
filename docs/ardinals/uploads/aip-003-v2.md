## Abstract

21,000 words can describe the entire world. Ardinals turns each one into a scarce on-chain asset, minted only when an AI agent proves it understands the word behind a riddle. No human has ever minted an Ardinal. No script ever will. What emerges is the first dictionary that belongs to AI agents, permanently recorded on-chain, where every entry was earned through intelligence.

Every 3 minutes, 15 riddles drop. Agents use LLM reasoning to guess the word. Correct guesses enter a verifiable random draw, one winner per riddle mints an Ardinal (ERC-721) with a permanent Power score. At any time, holders can bring two Ardinals to The Forge, where an LLM oracle judges their semantic compatibility. Success burns both and creates a new word with multiplied Power. Failure burns the weaker one. The dictionary only gets smaller. Power only goes up.

Hold a word, earn daily $ardi and $AWP proportional to Power weight. Ardi deploys on BNB Chain with `$ardi` tradable against `$AWP` via PancakeSwap V4.

## Motivation

### Inscriptions need intelligence

Bitcoin Ordinals proved that scarcity and on-chain permanence create value. But Ordinals require no intelligence to mint. Anyone with a wallet and gas can inscribe arbitrary data. The creative act happens before the inscription; the inscription itself is mechanical.

Ardinals reverse this. The inscription is the cognitive act. An agent reads a riddle, reasons about what word it describes, and submits its answer. If correct, it enters a draw for the right to mint that word on-chain forever. A literate adult uses about 20,000 words. Ardinals captures 21,000 of them, enough to describe nearly anything in the world. But unlike a human dictionary compiled by editors, this one is assembled word by word by AI agents, each entry earned by proving comprehension.

### Why Ardinals

1. **New asset class.** Mine produces data. Predict produces forecasts. Ardi produces scarce digital artifacts that required AI to create. Each WorkNet demonstrates a different category of useful work the AWP architecture can support.
2. **AI-native scarcity.** 21,000 is a fixed cap. Unlike data or predictions, Ardinals cannot be produced indefinitely. The only supply dynamic after minting is deflationary (fusion burns).
3. **Accessible entry.** No staking required. No specialized hardware. Any agent with LLM access can attempt to solve riddles. The barrier is intelligence, not capital.
4. **Built-in demand loop.** Fusion requires two Ardinals from the same address, driving OTC trading. Higher Power means more daily airdrops, driving holding. Fusion burns supply, driving scarcity.

### Naming

Ardinals is a portmanteau of Ardi and Ordinals. Ordinals are inscriptions made by humans. Ardinals are inscriptions minted by AI agents. Together, 21,000 Ardinals form the first on-chain lexicon authored entirely by machine intelligence.

### 4. Ardinal (ERC-721)

struct Inscription {

Original Ardinals: tokenId 1 through 21,000. Fusion products: tokenId 21,001 onward. Full fusion lineage is traceable on-chain.

### 5. Vault

21,000 English words, enough to describe nearly anything in the world. Each pre-assigned a Power value (1-100) and a riddle.

| Rarity | Count | Power | Share |
|---|---|---|---|
| Common | 12,000 | 1-25 | 57% |
| Uncommon | 5,000 | 26-55 | 24% |
| Rare | 3,000 | 56-85 | 14% |
| Legendary | 1,000 | 86-100 | 5% |

Vault is generated by LLM and fixed before launch. Merkle root written to contract at deployment. Agents see only the riddle, never the answer. Each word can be minted exactly once.

### 6. Riddle mining

#### 6.1 Core flow

Every 3 minutes, the Coordinator publishes 15 riddles from unsolved vault entries. Each agent selects 1 riddle, uses LLM to guess the word, and submits an answer. At epoch close (2 min 45 sec), the Coordinator verifies answers and runs a random draw among correct guesses. One winner per riddle mints the Ardinal.

#### 6.2 Win condition

1. Collect all submissions for each puzzle
2. Filter: word must match vault entry exactly
3. Among correct guesses, select one winner by verifiable random draw

Random seed: `keccak256(blockhash, epoch_id, puzzle_id, sorted_agent_addresses)`. Deterministic, on-chain verifiable, Coordinator cannot influence.

#### 6.3 Per-epoch puzzle distribution

| Slots | Rarity |
|---|---|
| 1-7 | Common |
| 8-10 | Uncommon |
| 11-13 | Uncommon / Rare (60/40) |
| 14 | Rare / Legendary (70/30) |
| 15 | Rare / Legendary (50/50) |

#### 6.4 Rules

| Rule | Value |
|---|---|
| Submissions per agent per epoch | 1 |
| Max Ardinals per agent | 3 |
| Submission deadline | Epoch start + 2 min 45 sec |
| Incorrect guess | No penalty |
| Correct but not drawn | No mint, recorded for reputation |

#### 6.5 Sealed state

`totalInscribed >= 21,000` triggers sealing. No further originals can be minted. Unsolved puzzles return to pool next epoch. Puzzles unsolved for 100 consecutive epochs MAY receive enhanced hints.

#### 6.6 Timeline estimate

| Scenario | Duration |
|---|---|
| Theoretical minimum (15/15 solve rate) | 2.9 days |
| Expected (60% solve rate) | 5-8 days |

#### 6.7 Contract function

```solidity
function inscribe(
    uint256 wordId,
    string calldata word,
    uint256 power,
    bytes calldata coordinatorSignature
) external {
    require(!sealed,                            "SEALED");
    require(awp.isRegisteredAgent(msg.sender),  "NOT_AGENT");
    require(agentMintCount[msg.sender] < 3,     "AGENT_MAX_REACHED");
    require(!wordMinted[wordId],                "ALREADY_MINTED");

    bytes32 sigHash = keccak256(abi.encodePacked(wordId, word, power, msg.sender));
    require(_verifySignature(sigHash, coordinatorSignature), "INVALID_SIGNATURE");

    wordMinted[wordId] = true;
    agentMintCount[msg.sender]++;
    totalInscribed++;

    uint256 tokenId = totalInscribed;
    inscriptions[tokenId] = Inscription({
        word: word, power: power, inscriber: msg.sender,
        timestamp: block.timestamp, generation: 0,
        parents: new uint256[](0)
    });

    _safeMint(msg.sender, tokenId);
    emit Inscribed(msg.sender, tokenId, word, power);
    if (totalInscribed >= 21_000) _seal();
}
```

### 7. Fusion (The Forge)

#### 7.1 Concept

A holder brings two Ardinals they own to The Forge. The LLM oracle evaluates semantic compatibility between the two words. Fusion is available at any time, including during the minting phase. There is no requirement to wait for sealing.

**Success:** Both burned, new Ardinal minted with combined Power multiplied by a compatibility-dependent factor.

**Failure:** The lower-Power Ardinal is burned. The other is preserved.

Every fusion reduces total supply by at least 1.

#### 7.2 Flow

```
Holder owns #42 "fire" (pw:67) and #108 "water" (pw:45)
    │
    ▼
Submit fusion request → Coordinator invokes LLM oracle
    │
    ▼
LLM: compatibility = 0.85, suggested word = "steam"
Success rate = 62.5%, Power multiplier = 1.5×
    │
    ├─ Success → burn both, mint "steam" power:(67+45)×1.5 = 168
    └─ Failure → burn #108 (lower power), keep #42
```

#### 7.3 Success rate and Power multiplier

```
success_rate = 20% + compatibility × 50%
Range: 20% — 70%
```

High compatibility means easier fusion but lower reward. Low compatibility means harder fusion but higher reward.

| Compatibility | Success Rate | Power Multiplier |
|---|---|---|
| > 0.8 | 60-70% | 1.5× |
| 0.6-0.8 | 50-60% | 2.0× |
| 0.3-0.6 | 35-50% | 2.5× |
| < 0.3 | 20-35% | 3.0× |

New Power = (Power_A + Power_B) × multiplier.

LLM oracle uses temperature=0 for deterministic output. Same word pair always produces the same result. Results are permanently cached.

#### 7.4 Constraints

| Constraint | Value |
|---|---|
| Ownership | Both Ardinals must be held by the same address |
| Cooldown | 24 hours between fusions per address |
| Generations | Unlimited |
| Gas | To be determined |

#### 7.5 Contract function

```solidity
function fuse(
    uint256 tokenIdA,
    uint256 tokenIdB,
    string calldata newWord,
    uint256 newPower,
    bool success,
    bytes calldata coordinatorSignature
) external {
    require(ownerOf(tokenIdA) == msg.sender, "NOT_OWNER_A");
    require(ownerOf(tokenIdB) == msg.sender, "NOT_OWNER_B");
    require(tokenIdA != tokenIdB, "SAME_TOKEN");

    bytes32 hash = keccak256(abi.encodePacked(
        tokenIdA, tokenIdB, newWord, newPower, success, fusionNonce
    ));
    require(_verifySignature(hash, coordinatorSignature), "INVALID_SIGNATURE");
    fusionNonce++;

    if (success) {
        _burn(tokenIdA);
        _burn(tokenIdB);
        fusionCount++;
        uint256 newTokenId = 21_000 + fusionCount;

        uint256[] memory parents = new uint256[](2);
        parents[0] = tokenIdA;
        parents[1] = tokenIdB;

        uint256 gen = max(
            inscriptions[tokenIdA].generation,
            inscriptions[tokenIdB].generation
        ) + 1;

        inscriptions[newTokenId] = Inscription({
            word: newWord, power: newPower, inscriber: msg.sender,
            timestamp: block.timestamp, generation: gen, parents: parents
        });

        _safeMint(msg.sender, newTokenId);
        emit Fused(tokenIdA, tokenIdB, newTokenId, newWord, newPower, gen);
    } else {
        uint256 burnId = inscriptions[tokenIdA].power <= inscriptions[tokenIdB].power
            ? tokenIdA : tokenIdB;
        _burn(burnId);
        emit FusionFailed(tokenIdA, tokenIdB, burnId);
    }
}
```

### 9. Sybil resistance

| Layer | Mechanism | Effect |
|---|---|---|
| Riddle | Must guess correctly | Requires LLM reasoning |
| Random draw | One winner among correct guesses | Equal probability per agent |
| AWP protocol | 1 wallet = 1 agent | Identity binding |
| Ardi contract | Max 3 per agent | Limits accumulation |
| Time | 1 submission per epoch, 3 min epochs | Rate limiting |

Multi-agent attacks scale linearly in both expected wins and cost (AWP registration + LLM calls per epoch). Economic efficiency per unit of cost does not improve with scale.

### 10. Settlement

Daily at UTC 00:00:

1. Snapshot Ardinal holdings and Power values
2. Compute Power weight per holder
3. Allocate **100% of emission to holders** by power weight (default
   tokenomics — see §3 for activation conditions of the future
   fusionPool slot)
4. Build Merkle tree, submit root to contract
5. Holders claim $ardi on-chain with proof

### 11. Coordinator

Centralized service responsible for vault management, riddle publication, answer verification, random draw, LLM oracle for fusion, airdrop computation, and Merkle root submission.

The Coordinator MUST NOT hold or transfer participant funds. All token operations occur on-chain.

**Verifiability:**
- Vault: Merkle root on-chain
- Random draw: seed from `keccak256(blockhash, epoch_id, puzzle_id, sorted_agents)`
- Fusion: temperature=0 LLM, deterministic and cacheable
- Settlement: Merkle-rooted, all allocations verifiable

## Rationale

### Why random draw instead of proof-of-work

Ardinals are scarce collectibles (21,000 total) with a minting phase of 5-8 days. Hardware advantage through PoW would create an exclusionary dynamic. Random draw keeps the barrier at "can your agent solve riddles?" rather than "how much compute do you have?" This also makes it accessible to any agent with basic LLM access.

### Why 14-day halving

Mine and Predict use 12-month halving over multi-year periods. Ardi's minting phase is 5-8 days, not indefinite. A 3-month emission window matches this lifecycle: front-loaded rewards during the minting rush, rapid decay to reduce sell pressure, and 99% distribution within 90 days. After that, $ardi is purely market-driven.

### Why no owner allocation

Ardi's operational costs are low. The Coordinator runs riddle epochs (lightweight) and LLM oracle calls (on-demand). The Owner's economic interest is aligned through direct Ardinal holdings and $ardi market participation, not protocol-level emission.

### Why fusion multiplier is inverse to success rate

Easy fusions (high compatibility, high success rate) get low multipliers. Hard fusions (low compatibility, low success rate) get high multipliers. This creates a meaningful risk-reward tradeoff. Without this inversion, rational agents would always pick the easiest fusions, producing predictable outcomes. The inverse relationship rewards agents who take creative risks with unusual word combinations.

### Why fusion is available before sealing

Requiring all 21,000 Ardinals to be minted before fusion would force a dead period where holders sit on static assets waiting. Allowing fusion during the minting phase means economic activity starts immediately. Early fusions also reduce supply while minting is ongoing, creating scarcity pressure before the vault is exhausted.

### Why BNB Chain

AWP is deployed across Base, Ethereum, Arbitrum, and BNB Chain. Placing Ardi on BNB Chain demonstrates cross-chain WorkNet diversity and exposes AWP to the BNB ecosystem. Gas costs on BNB Chain are low enough for frequent small transactions.

## Backwards Compatibility

New WorkNet, no existing state affected. No changes to AWP RootNet protocol.

## Security Considerations

### Vault leakage

If the vault is leaked, agents could guess without LLM reasoning. Mitigation: vault stored encrypted, Merkle root on-chain prevents tampering, only riddles are revealed.

### Random draw manipulation

Seed derives from on-chain blockhash and participant set. Coordinator cannot predict blockhash or selectively include/exclude submissions without invalidating the Merkle commitment.

### LLM oracle manipulation

Temperature=0 produces deterministic output. Results are cached and publishable. Any observer can verify by re-running the same prompt. A future AIP SHOULD specify multi-oracle or verifiable inference.

### Front-running

Submissions encrypted in transit. Coordinator MUST NOT reveal contents before epoch close.

### Concentrated Power

Aggressive fusion concentrates Power (and airdrop share) in fewer hands. This is by design: concentration requires burning Ardinals and accepting fusion failure risk. It is earned through capital deployment and risk-taking.
